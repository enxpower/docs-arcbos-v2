3A. Alpha / Beta 阶段能力描述（非技术版）

本章节用于说明 Alpha 与 Beta 阶段完成后的实际能力状态。

如与正式技术验收标准存在差异，以技术验收标准为准。

3A.1 Alpha 阶段能力描述：机器能被安全控制

Alpha 阶段的核心能力是：

Alpha Core Capability
├─ 上电 / 启动 / 可调试
├─ 工控机 + 底盘控制器 + 传感器 + 通信链路协同
├─ 上位机或遥控器控制
│  ├─ 前进
│  ├─ 后退
│  ├─ 转向
│  └─ 停车
├─ 基础状态反馈
│  ├─ 底盘状态
│  ├─ 传感器数据
│  ├─ 定位数据
│  ├─ 通信状态
│  └─ 故障状态
├─ 安全逻辑
│  ├─ 急停
│  ├─ 人工接管
│  └─ 基础安全停车
├─ 异常优先进入安全状态
├─ 基础日志
└─ 代码 / 配置 / 接口 / 部署方式可交付、可接管

Alpha 阶段不代表：

Alpha Exclusions
├─ 不代表自主除雪
├─ 不代表自动规划路线
├─ 不代表无人值守运行
├─ 不代表完整避障能力
├─ 不代表产品销售状态
└─ 不代表最终安全认证或法规认证完成

通俗判断标准：人可以安全地控制 SnowBot，机器能跑、能停、能反馈、能接管、能记录问题。

3A.2 Beta 阶段能力描述：机器能按人定义的路线完成基础任务

Beta 阶段的核心能力是：

Beta Core Capability
├─ 固定场地路线定义
│  ├─ 录制路线
│  └─ 定义任务边界
├─ 路线管理
│  ├─ 保存
│  ├─ 加载
│  └─ 重复执行
├─ 位置与状态判断
│  ├─ 定位
│  ├─ 里程计
│  └─ 传感器数据
├─ 低速路径跟踪
├─ 基础障碍检测
│  └─ 优先安全停车
├─ 可选绕障
│  ├─ 必须在安全边界内
│  └─ 必须能回归原路线
├─ 异常降级 / 停车
│  ├─ 定位不可靠
│  ├─ 传感器异常
│  ├─ 通信异常
│  └─ 底盘异常
├─ 人工随时接管
├─ 日志 / 测试记录 / 问题记录可复盘
└─ 可用于工程演示 / 内部测试 / 客户初步交流 / 融资展示

Beta 阶段不代表：

Beta Exclusions
├─ 不代表任意场地自主运行
├─ 不代表无人值守商业运营
├─ 不代表自动理解复杂环境
├─ 不代表全区域覆盖路径规划
├─ 不代表自动回充或自动返航
├─ 不代表无人工监督处理所有异常
├─ 不代表量产交付或客户正式运营状态
└─ 不代表最终安全认证或法规认证完成

通俗判断标准：人定义路线，机器人能按路线自动跑一遍，遇到不确定情况能停下来，问题可被记录和复盘，ARCBOS 可接手继续开发。

3A.3 Alpha 与 Beta 的通俗区别

Alpha：人控制机器，证明机器可控（能安全跑起来）。
Beta ：人定义任务，机器执行任务，证明任务可闭环（能按路线完成基础任务）。

Alpha / Beta 阶段均属于工程样机阶段，不属于最终商业化产品阶段。

→ 日志记录 / 数据复盘

4F. 扩展性架构原则 / Extension Architecture Principles

本阶段不开发 Gamma / Delta / Fleet。但 Alpha/Beta 的软件结构必须预留以下四条“扩展缝”，使后续阶段为叠加而非重写。预留扩展缝不增加当前阶段开发范围，仅要求当前接口与模块边界按可扩展方式定义。

Extension Seams (零额外范围，服务于后续阶段)
├─ 定位输出为抽象 Pose 源，而非绑死 RTK
│   └─ Delta 可在同一接口背后增加 LiDAR / 视觉里程计兑底
├─ 路线为带版本号的数据契约，而非临时文件
│   └─ Gamma 覆盖规划输出相同路线 Schema
├─ 决策/任务为状态机，置于单一仲裁器之后
│   └─ Fleet 只需增加远程指令源，其余不变
└─ 每个节点以统一信息体上报 health + heartbeat
    └─ 无人值守仅为“谁响应 health 总线”的扩展

4F.1 安全仲裁器 / Safety Arbiter

第 8 节定义了 6 级安全优先级，但必须由单一组件强制执行。

Safety Arbiter
├─ 为最终 motion-enable 信号的唯一发布者
├─ 汇聚所有停车 / 接管 / 降级来源并按优先级仒裁
├─ 发布 /snowbot/safety/arbitration
├─ 不允许任何其他节点绕过仲裁器直接使能运动
└─ 为 M1 交付物（架构与接口冻结）

8. 安全优先级

系统安全优先级必须明确。

优先级安全/控制来源原则
1物理急停最高优先级，任何自动控制不得覆盖
2遥控器接管 / 人工停车人工接管优先于自动控制
3底盘底层保护底层硬件保护可拒绝危险指令
4软件安全停车上位机可触发安全停车或降级
5自动驾驶控制只在安全条件满足时执行
6任务执行策略不得高于任何安全逻辑

任何情况下，自动驾驶控制不得高于急停、遥控器接管和底盘底层保护。

系统必须覆盖以下异常，并对每类异常定义检测条件、处理动作、是否停车、是否允许恢复、是否记录日志、是否需要人工确认。

异常类别异常项默认处理原则是否需记录日志

定位异常RTK-GPS 数据超时停车或进入安全状态是
姿态/里程异常IMU 数据异常降级或停车是
姿态/里程异常odometry 数据异常降级或停车是
感知异常lidar 数据丢失停车或进入安全状态是
感知异常lidar 数据超时停车或进入安全状态是
通信异常CAN 通信超时停车或进入安全状态是
底盘异常底盘故障反馈停车或等待人工处理是
上位机异常上位机节点异常停车或进入安全状态是
任务异常路线文件加载失败禁止启动任务是
控制异常控制输出异常停车或进入安全状态是
障碍异常障碍物进入安全区优先安全停车是
人工介入人工接管自动控制释放是
急停急停触发所有自动控制失效，进入急停状态是

异常处理总原则：

不确定时停车。
通信丢失时停车。
人工接管时自动控制释放。
急停触发后不得自动恢复运行。

9E. 接口数据契约 / Interface Data Contracts

9A 仅定义 topic / CAN 名称。本节定义其载荷与语义。所有契约（ROS2 .msg、CAN DBC、route schema、health schema、fault_code_table）必须在 M1 冻结，并存放于 contracts/ 顶层目录，作为 snowbot-autonomy 与 snowbot-embedded 双方共同引用的唯一真源。名称无 Schema 不视为接口冻结。

9E.1 定位健康 / Localization Health

所有“定位异常 → 降级或停车”判定必须基于以下可量化字段，而非主观判断。

/snowbot/localization/health
├─ fix_type        : NONE | SINGLE | FLOAT | FIX
├─ horizontal_std_m: float   (路线模式下 > 阈值即停车，阈值见 11A)
├─ age_s           : float   (数据时延，超阈值停车)
├─ heading_std_deg : float
└─ status          : OK | DEGRADED | INVALID   (计算值，非原始值)

9E.2 路线文件 Schema / Route File Schema

路线文件必须带版本号，并定义安全走廊宽度，使“绕障必须在安全边界内并能回归路线”可被验收。Gamma 覆盖规划须输出相同 Schema。

route_file v1 (versioned)
├─ schema_version
├─ frame_id            (固定场地局部坐标系)
├─ recorded_at, site_id
└─ waypoints[]
   ├─ x, y, heading
   ├─ target_speed
   ├─ corridor_half_width   ← 定义“安全边界”
   └─ segment_flags         (如 work_unit_on / off)

9E.3 CAN 报文语义 / CAN Message Semantics

9A.2 每条 CAN 报文必须定义：ID、周期、字节布局、单位、有效范围、超时行为。

CAN Contract (每条报文必须包含)
├─ CAN ID
├─ cycle_time_ms
├─ byte_layout / signals
├─ unit + valid_range
└─ timeout_behavior

关键安全要求 —— 指令心跳 / 看门狗：

├─ MCU 若在 N ms 内未收到 MOTION_CMD，底盘必须自主安全停车
├─ 该 fail-safe 必须实现在 MCU 侧，而非仅在 IPC 侧
├─ 即 IPC 崩溃 / 断链时，机器仍必须停止
└─ 具体实现以嵌入式主需求文档为准，本文件交叉引用并要求双方冻结

9F. 时间同步与坐标系 / Time Sync & Coordinate Frames

Time Synchronization
├─ RTK / IMU / odom / lidar 融合要求统一时钟
├─ 采用 PTP 或 chrony；如有硬件 PPS 优先使用
├─ 每个传感器须声明时间戳来源
└─ 时钟不同步会静默污染位姿融合，必须在 M1 明确

Coordinate Frames
├─ 采用 REP-103 / REP-105 约定 (ENU; map → odom → base_link)
├─ 主要传感器与底盘 TF 关系须文档化
└─ 避免每个供应商各自的坐标系约定
