SnowBot Alpha/Beta 技术节点与验收付款条件 v1.2

适用项目：SnowBot Alpha补齐 + Beta闭环

文件性质：合同附件 / 技术节点 / 验收条件 / 付款触发依据

对应文件：附件一《SnowBot Alpha/Beta 导航与VCU控制技术基线包》

1. 文件定位

本附件用于定义 P0-P5 付款节点对应的技术验收条件和付款触发依据。

附件一规定技术范围和最低技术要求；本附件仅规定各付款节点应达到的状态、应提交的证据和不予通过的情形。

除 P0 预付款外，后续节点不得仅以口头说明、视频演示、截图、远程展示或乙方电脑可运行作为验收依据。

2. 节点总表

P110%六电机桌面调通六个电机可在桌面测试条件下启动、停止并形成记录测试记录、参数/协议、日志、GitHub提交

P320%预设路线或A/B点导航固定场地低速执行路线或A点至B点导航，异常可降级或停车目标/路线文件、导航日志、测试报告、版本记录

3. P0 预付款

P0为项目启动款，不构成技术成果验收。

P0付款条件：

  • 主合同及附件签署完成。
  • 乙方确认项目启动。
  • 乙方确认使用甲方指定 GitHub 仓库和 control-work 分支。
  • 双方确认初始工作计划和沟通机制。

4. P1 六电机桌面调通

P1目标：在桌面测试条件下完成六个电机的基础控制验证。

六个电机包括：左履带电机、右履带电机、除雪电机、抬升电机、旋转电机、俯仰电机。

P1最低验收条件：

  • 六个电机可按指令启动、停止。
  • 左、右履带电机可完成基础差速控制或等效转向控制验证。
  • 除雪、抬升、旋转、俯仰电机可完成基础转动或动作控制验证。
  • 控制参数、状态反馈、故障反馈具备初步记录。
  • 急停或软件停车逻辑完成基础验证。

P1交付依据：

  • VCU规格型号、接口和接入建议。
  • VCU配置或参数备份；如已形成。
  • CAN协议初版或CAN ID表初版。
  • 六电机桌面测试记录。
  • 运行日志或CAN日志。
  • 问题清单。
  • GitHub control-work 分支对应提交记录。

以下情形不构成P1通过：六电机未全部完成基础验证；缺少参数、协议、测试记录或日志；测试结果无法对应GitHub提交；急停或停车逻辑完全未验证。

5. P2 装机运动控制

P2目标：电机装机后，机器可通过遥控器或上位机实现基础运动和作业电机动作。

P2最低验收条件：

  • 机器可完成前进、后退、左转、右转、停车。
  • 遥控器可操作底盘运动。
  • 遥控器或上位机可操作除雪、抬升、旋转、俯仰电机转动；以硬件和安全条件满足为前提。
  • 急停、软件停车、人工接管链路有效。
  • 底盘状态、作业机构状态、通信状态、故障状态可记录或反馈。

P2交付依据：

  • 装机后的VCU配置和参数备份。
  • CAN协议或CAN ID表更新版。
  • IO表、接线图。
  • 底盘运动测试记录。
  • 作业电机测试记录。
  • 急停、软件停车、人工接管测试记录。
  • 运行日志或CAN日志。
  • 问题清单。
  • GitHub control-work 分支对应提交记录。

以下情形不构成P2通过：基础运动不完整；急停、软件停车或人工接管任一关键链路失效；缺少接线图、IO表、协议或日志；装机测试与提交版本无法对应。

6. P3 预设路线或A/B点导航

P3目标：在双方确认的固定测试场地、低速、安全条件下，实现预设路线行走，或从指定A点至指定B点的基础导航演示。

P3最低验收条件：

  • 目标点、路线或等效任务边界可配置。
  • 机器可在固定测试场地低速条件下执行基础路径跟踪或A点至B点导航演示。
  • RTK/GPS、IMU、Odometry、Lidar、超声波或约定传感器数据可接入并用于测试。
  • VCU可稳定执行导航系统下发的基础运动指令。
  • 任务状态至少包括未启动、运行中、暂停、停止、完成、故障或异常。
  • 定位异常、传感器异常、CAN异常、底盘异常或人工接管时，可降级或停车。

P3交付依据：

  • 目标点或路线配置文件。
  • 导航源码和配置文件。
  • 传感器数据接入说明。
  • 坐标系、TF和时序说明。
  • VCU控制接口映射说明。
  • 运行日志、传感器日志或CAN日志。
  • P3测试报告。
  • 问题清单。
  • GitHub control-work 分支对应提交记录。

以下情形不构成P3通过：仅人工遥控完成路线；无目标点、路线文件或等效任务配置记录；导航演示无法与代码、配置、日志和版本记录对应；异常时无降级、停车或人工接管能力；只能在乙方环境中运行。

7. P4 合同范围内任务完成

P4目标：完成附件一约定的 Alpha/Beta 阶段主要技术任务，形成可演示、可复现、可接管、可继续开发的工程交付状态。

P4最低验收条件：

  • VCU控制、六电机控制、遥控控制、上位机控制、急停、软件停车、人工接管和安全停车链路达到附件一要求。
  • 固定场地低速A点至B点导航或预设路线行走达到附件一要求。
  • 基础障碍检测可触发停车；条件满足时可尝试有限绕行；条件不满足时安全停车。
  • 传感器、VCU、导航、任务状态机、安全仲裁、日志系统形成基本闭环。
  • 源码、配置、接口、协议、接线图、IO表、参数、日志、测试数据、测试报告和问题清单完整提交。
  • GitHub control-work 分支中的代码、配置、文档与测试报告对应。
  • 不存在影响甲方部署、复现、接管或继续开发的阻塞性缺陷。

P4交付依据为完整 Release Package，至少包括：源码、VCU源码/配置/参数备份、可随VCU交付的内置履带差速代码、导航源码与配置、CAN协议或CAN ID表、ROS2接口清单、IO表、接线图、故障码表、VCU型号/BOM、导航传感器规格型号/接口/安装建议、部署/构建/运行说明、测试脚本或测试流程、Alpha测试报告、Beta测试报告、系统联调测试报告、运行日志、CAN日志、传感器日志、问题清单、版本说明、交接说明、第三方组件和硬件随附能力披露清单。

以下情形不构成P4通过：演示通过但工程交付不完整；甲方无法部署、复现或接管；关键安全链路失效；A点/B点导航、障碍停车或有限绕行未达到附件一最低要求；使用未披露或未经确认的第三方组件；交付内容与GitHub版本、测试报告、日志无法对应；存在其他阻塞性缺陷。

8. P5 质保金

P5为质保金节点，用于保证乙方在交付后质保期内对合同范围内问题进行配合整改。

P5支付条件：

  • P4已通过验收。
  • 质保期届满；质保期以主合同约定为准。
  • 质保期内未发生未解决的阻塞性缺陷。
  • 乙方已完成质保期内应由乙方负责的缺陷修复、说明或整改。
  • 甲方确认无影响部署、复现、接管或继续开发的重大未结问题。

以下情形下，质保金不得自动释放，除非甲方书面确认：存在未解决阻塞性缺陷；乙方未补齐关键交付文件；出现第三方组件、授权、费用、使用限制或知识产权争议；交付内容无法复现或无法由甲方接管；乙方未完成质保期内应承担的缺陷整改。

9. 验收流程

每一付款节点的验收流程如下：

节点验收流程
├─ 1. 乙方提交节点验收申请
├─ 2. 乙方提交对应交付物和版本记录
├─ 3. 甲方进行功能测试和资料核查
├─ 4. 如存在问题，乙方整改并重新提交
├─ 5. 甲方书面确认节点通过
└─ 6. 按主合同触发对应付款

甲方可根据项目情况采用现场测试、视频会议测试、远程日志审查、GitHub版本核查、实物测试或书面资料审查等方式进行验收。

乙方不得以“已演示过”“视频中可见”“乙方电脑可运行”作为拒绝补交工程文件或拒绝整改的理由。

10. 通用验收原则

每个节点验收应同时满足：功能达到节点最低要求，交付物完整，测试数据和测试报告可对应，GitHub提交记录清晰，版本可追踪，甲方可复现、可接管或可继续开发，不存在影响下一节点推进的阻塞性缺陷。

如某一功能受甲方未完成机械安装、电气供电、硬件取得、测试场地或安全条件影响，乙方应及时书面说明影响原因、依赖条件和替代验证方式。经甲方确认后，可采用桌面测试、仿真测试、局部联调或分阶段验收方式处理。

11. 阻塞性缺陷

以下情形属于阻塞性缺陷，包括但不限于：

  • 缺少源码、配置、协议、接线图、IO表、参数备份或部署说明。
  • 仅能在乙方电脑、乙方账号或乙方远程环境运行。
  • 安全停车、急停、人工接管任一关键链路失效。
  • 底盘基础运动控制无法稳定执行。
  • 六电机控制范围内任一应交付控制功能无法完成基础联调，且影响下一阶段推进。
  • A点/B点或路线目标配置、导航演示、障碍停车/有限绕行无法完成最低闭环。
  • 交付内容与测试报告、日志、版本记录无法对应。
  • 使用未披露或未经甲方确认的第三方组件。
  • 代码、注释、配置、接口、日志字段或工程命名不符合 UTF-8 与英文要求，且影响构建、运行、维护、交接或二次开发。
  • 存在影响甲方商业化使用、二次开发、委托第三方继续开发或知识产权转让的未披露限制。

存在阻塞性缺陷的节点，不应视为验收通过。

12. 与其他文件的关系

附件一规定技术范围、技术要求、工程边界和最低验收基线；本附件规定技术节点、验收条件和付款触发条件；附件三规定源码、配置、协议、资料、测试数据和交付物清单；主合同规定合同金额、付款安排、知识产权、保密、违约和争议解决。

如本附件与主合同在付款金额、付款比例或付款时间上存在不一致，以主合同为准。如本附件与附件一在技术要求上存在不一致，以附件一较明确、较高且不扩大合同范围的技术要求为准。
