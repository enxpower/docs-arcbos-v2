2. 三包制总图

SnowBot电气系统采用三包制外包 + ARCBOS总控。

SnowBot电气分包结构

A包：电池能源包
└─ 安全、稳定、可诊断地输出48V主电源。

B包：整车电控总包
└─ 让机器人安全地动起来、除起来、停下来、报故障。

C包：产品化电气工程包
└─ 把Alpha跑通的系统产品化，包含Dock、正式线束、低压、EMC。

I包：ARCBOS总架构包
└─ 定义系统、控制接口、组织集成、执行验收、决定冻结。

Alpha阶段启动A包和B包。Beta阶段在A包和B包边界稳定后启动C包。I包贯穿始终。

3. 第一性原理下的ARCBOS职责

SnowBot电气系统本质上只解决四件事：

SnowBot电气系统四件事

储能与安全供电
└─ A包主责，I包定义边界并验收。

动力执行与运动控制
└─ B包主责，I包定义状态机、安全边界和验收。

补能与产品化电气工程
└─ C包主责，I包定义互锁、接口和冻结标准。

整机集成与验证
└─ I包主责，不能外包。

外包越多，接口越多。ARCBOS现阶段的最佳实践不是细拆供应商，而是减少接口、明确责任、牢牢掌握架构和验收。

6. ARCBOS对B包的控制点

B包控制点

动力边界
├─ 除雪总成2.5–3.5kW连续
├─ 除雪总成5–6kW峰值
├─ 不冻结5kW连续功率
├─ 履带1.0–1.5kW/侧连续目标
└─ 履带2.0–2.5kW/侧峰值目标

控制边界
├─ VCU状态机
├─ 堵转检测
├─ 反转清堵
├─ 负载仲裁
├─ 故障码
└─ 日志

安全边界
├─ E_STOP最高优先级
├─ CHARGING禁止履带和除雪动作
├─ SERVICE禁止自动动作
├─ 满电下坡硬规则
└─ NAV/COMMS降级动作

B包不能把系统做成黑盒。ARCBOS必须拿到VCU状态机图、CAN协议、IO点表、故障码表、负载仲裁策略和测试数据。

9. 系统级安全边界

以下边界由ARCBOS定义，不由供应商决定。

系统级安全边界

电池安全
├─ 低温硬件禁充
├─ 过流保护
├─ 接触器异常处理
└─ 带载拔出阻止

运动安全
├─ E_STOP
├─ SERVICE禁止自动动作
├─ CHARGING禁止运动
├─ 堵转两次失败进入FAULT_STOP
└─ 通信丢失不得维持危险动作

充电安全
├─ C1–C12互锁
├─ 未到位不带电
├─ 接触电阻异常不快充
└─ 退出前降流到0

场景安全
├─ 满电下坡硬规则
├─ NAV_LOST停止新区域作业
├─ COMMS_LOST不进入新区域
└─ 3cm以上硬冰不承诺处理

15. 总体验收标准

SnowBot电气系统进入Beta冻结前，必须满足：

  1. A/B/C包接口文件完整。
  2. BMS CAN、VCU CAN、电机控制器CAN、充电接口清楚。
  3. 主回路保护协调有初步依据。
  4. 除雪总成堵转逻辑可测试。
  5. 履带坡道和原地转向测试完成。
  6. 满电下坡策略可执行。
  7. NAV/COMMS降级动作可执行。
  8. 充电互锁C1–C12完整。
  9. 正式线束图和端子表完整。
  10. 一票否决项全部关闭或有明确整改计划。

20. 四包自查结论

本轮核查后，ABCI四份文档必须满足以下判断。

四包自查结论

A包
├─ 不只是电芯/BMS
├─ 必须覆盖电池结构、主回路、BMS、热管理、保护协调和认证资料
└─ 对整机负责的是48V能源安全底座

B包
├─ 不只是电机/控制器
├─ 必须覆盖除雪、履带、VCU、状态机、负载仲裁、硬冰边界、满电下坡和故障码
└─ 对整机负责的是安全运动与安全除雪闭环

C包
├─ 不只是线束/Dock
├─ 必须覆盖正式线束、低压配电、充电Dock、尾部触点、EMC、装配和产品化验证准备
└─ 对整机负责的是产品化电气工程落地

I包
├─ 不做具体零件设计
├─ 必须掌握系统定义、接口控制、安全边界、验收和冻结
└─ 对整机负责的是防止供应商各自为政

24.4 接口变更审批流程

接口变更审批流程

R0-R1阶段
├─ 供应商可提出变更建议
├─ I包记录变更
└─ 安全相关变更必须立即评审

R2评审后
├─ 任何接口变更必须提交接口变更申请
├─ I包评估影响
├─ 相关包确认影响
├─ 安全相关变更必须Founder/总架构批准
└─ 批准后更新ICD版本

Beta冻结后
├─ 原则上不允许接口变更
├─ 必须变更时建立ECR/ECN记录
└─ 重新执行受影响测试

接口变更申请最少包含：变更编号、提出方、变更描述、影响包、安全影响、线束影响、测试影响、批准人、批准日期、关闭证明。
